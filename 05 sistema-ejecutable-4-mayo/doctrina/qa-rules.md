# Reglas QA + 7 reglas de fidelidad

> Las reglas que cada output del cluster debe cumplir antes de publicar. Si alguna falla, **bloquea el deploy**.

---

## Las 12 reglas QA por página (literal del original §7-§18)

Cada URL del cluster pasa por estas 12 reglas antes de publicarse. Si las 12 son **SÍ**, la página está `Ready to publish`.

| # | Regla | Qué valida | Implementación |
|---|---|---|---|
| 1 | URL QA | URL sigue patrón doctrinal del page type | `scripts/qa-check.ts` test 1 |
| 2 | Page Type QA | H1 + estructura + schema vs spec del page type | test 2 |
| 3 | One service only | LBS habla de UN solo servicio | test 3 |
| 4 | One Main City only | URLs locales solo `/madrid/...`, NO `/sevilla/...` | test 4 |
| 5 | Local Coverage QA | LCAs presentes en copy + en `areaServed` schema | test 5 |
| 6 | No fake location | NAP coherente (HTML + schema + footer + página contacto) | test 6 |
| 7 | Metadata QA | H1 + meta title + meta description coherentes | test 7 |
| 8 | Word count QA | mínimos por page type cumplidos (HP 900-1300, SO 850-1000, ...) | test 8 |
| 9 | Schema QA | JSON-LD validado por page type | test 9 |
| 10 | Enlaces internos QA | matriz linking interno cumplida (ver `doctrina/linking.md`) | test 10 |
| 11 | Canibalización QA | 0 pares de URLs solapando intención | test 11 |
| 12 | CTA QA | CTA adaptado al page type según funnel position | test 12 |

### Final Publish Gate

```
Final Publish Gate = Test 1 ∧ Test 2 ∧ ... ∧ Test 12
```

Si Final Publish Gate = `true`, la página puede publicarse. Si `false`, bloquea Paso 18.

## Auditorías Paso 18 (35 criterios literales)

Ejecutadas justo antes del deploy. Si algún criterio falla, deploy se aborta.

### 18.1 Auditoría Técnica (16 criterios)

```
1.  Lighthouse desktop ≥ 90
2.  Lighthouse mobile ≥ 85
3.  Core Web Vitals OK (LCP, FID, CLS)
4.  sitemap.xml válido + accesible en /sitemap-index.xml
5.  robots.txt OK (no bloquea páginas indexables)
6.  404 personalizada
7.  Trailing slash consistente (todas terminan en `/`)
8.  Canonical en cada página
9.  hreflang correcto (si multi-idioma)
10. Favicon presente (16x16, 32x32, apple-touch)
11. OG tags (og:title, og:description, og:image, og:url)
12. Twitter Cards (twitter:card, twitter:title, twitter:description)
13. Meta viewport
14. Charset UTF-8
15. Structured data validado (JSON-LD parsea OK)
16. Imágenes con alt
```

### 18.2 Auditoría de Contenido (8 criterios)

```
1. Word counts cumplen mínimos por page type
2. NAP coherente en HTML + schema + footer + contacto
3. LCAs presentes en copy + areaServed
4. CTAs por page type según funnel
5. 0 menciones de ciudades benchmark como zona operativa
6. GAs enlazan a LBS padre (no SO)
7. Trust block en HP
8. Contacto inline en HP
```

### 18.3 Auditoría Visual (5 criterios)

```
1. Tipografía coherente (1-2 families max)
2. Paleta aplicada (tokens del cliente o default)
3. Spacing consistente (8px-grid)
4. CTA destacado visualmente
5. Responsive 375px - 1920px sin desbordamientos
```

### 18.4 Auditoría Rastros IA (6 criterios)

```
1. Sin frases robóticas tipo "como modelo de lenguaje", "es importante destacar"
2. Sin emojis innecesarios
3. Prosa natural (no bullet points encadenados sin contexto)
4. Sin listas irregulares (mezclas - y * y números)
5. Limpieza de placeholders generados ("Lorem ipsum", "TODO:")
6. Commits sin frases tipo "Generated by Claude"
```

## Las 7 reglas de fidelidad al original (Convenciones §3)

Reglas operativas para evitar inventar doctrina del manual original.

### Regla 1 — Verificación bidireccional antes de declarar un output

Pregunta antes de añadir un output a un paso: **¿el original §X-Outputs lo lista?**

- **Sí** → output nativo del paso, OK.
- **No (pero se usa aquí)** → input heredado, va a "Info heredada", **NO** a outputs.
- **No (ni se usa)** → no debería estar.

**Caso real:** los slugs (`primary-category-slug`, `service-slugs`, etc.) se DECIDEN en Paso 3 (intake §4), se APLICAN en Pasos 2/4/5. Los paso 2/4/5 los reciben heredados, NO los producen.

### Regla 2 — Los 7 patrones de invención prohibidos

#### 2.1 Outputs inventados

NO promocionar a output del paso algo que el original no lista.

> **Caso real:** `5.1 Primary Service` y `5.2 Brand tone` declarados como outputs del Paso 5 cuando el original solo los lista en intake del Paso 1.

#### 2.2 Outputs robados de otro paso

NO declarar como output nativo algo que en el original es output de otro paso anterior.

> **Caso real:** `2.2 Primary Category Slug` en Paso 2 cuando los slugs son output del Paso 3.

#### 2.3 Doctrina inventada

NO añadir afirmaciones cuantitativas o cualitativas no presentes en el original.

> **Casos reales detectados:**
> - `S_efectiva` como variable doctrinal (original solo usa `S`)
> - "29 filas / 580 celdas operativas" (original §27.3 dice 28 URLs)
> - Fórmula `E × (1 + S + A + G×S)` para Approved Expansion (no en original)
> - Distribución porcentual anchors `~15% / ~25% / ~10%` (original no asigna %)
> - Threshold `≥3 enlaces contextuales por página` (original no declara mínimo)
> - 6ª categoría anchor "Local entity" (original lista 5)

#### 2.4 Doctrina cambiada

NO alterar literalmente lo que dice el original.

> **Casos reales:**
> - "AC añadido a outbound de Homepage" — original §8 / §20 solo cita SO + GeoHub + contacto
> - Schema Homepage con `FAQPage + Speakable` añadidos — original §16 dice solo `Organization, WebSite, LocalBusiness`
> - Formato IDs `SO-1` cuando el original §16 usa `SO-001`
> - Conteo bloques estructurales modificados (SO 11 vs 12, LBS 12 vs 13, AC 11 vs 12, GA 10 vs 11)

#### 2.5 Doctrina omitida

NO portar contenido literal del original al sistema operativo.

> **Caso real (el peor):** Paso 5 omitió las 18 secciones doctrinales del original — los 6 "Ejemplo rellenado" + 6 "Ejemplos incorrectos" + 6 "Regla final" por page type.

#### 2.6 Cambios silenciosos

NO modificar el original sin advertirlo. Tres tipos:

- **Erratas "arregladas":** original `Target long-tail llaveword` → portado `Target long-tail keyword`. **NO arreglar.**
- **Ambigüedades resueltas:** original §30.3 dice `areaServed: 4 LCAs ejemplo` → portado dice `areaServed: TODAS las 10 LCAs`. **NO resolver — flagear.**
- **Convenciones cambiadas:** formato IDs, casing, separadores. **NO cambiar.**

#### 2.7 Datos del cliente fabricados

NO añadir información concreta del cliente que el original no contiene.

> **Caso real:** Paso 6 portado tenía `streetAddress: "Calle Rafael Calvo 12"` cuando el original solo dice "Madrid".

> **Excepción NAP físico (mayo 2026):** se permite enriquecer ejemplo con NAP completo cuando es más realista que el original Y el operador lo aprueba explícitamente. Documentar en commit.

### Regla 3 — Qué SÍ está permitido añadir

Para evitar el otro extremo:

- Numeración `Paso.X` (1.1-1.14, 2.1-2.14, etc.) — diseño del esqueleto
- Tablas "Info heredada" — trazabilidad
- Cross-refs `Paso-NN N.X` — enlaces entre pasos
- Reformulación estructural prosa→tabla manteniendo contenido literal
- "Errores que previene" derivados doctrinalmente

### Regla 4 — Protocolo antes de portar un paso

```
1. Leer el original COMPLETO (no solo secciones que crees que aplican)
2. Listar outputs nativos del paso según §28-§30 del original
3. Identificar el intake del paso (§4 "Lo que tienes que rellenar")
4. Identificar secciones doctrinales por sub-tema (Ejemplo / Ejemplo incorrecto / Regla final)
```

### Regla 5 — Cuando el original tiene ambigüedad

```
1. NO resolver silenciosamente
2. Documentar la ambigüedad con `> **Nota de fidelidad:**`
3. Decidir en consulta con el operador
4. La opción no elegida queda referenciada con cross-ref pero NO desplazada
```

### Regla 6 — Cuando el original es escueto

Original dice `1 low → 5 high` sin detallar.

```
- Portar literal: si el original dice `1 low → 5 high`, escribir exactamente eso
- NO inventar 5 niveles narrativos
- Si el operador necesita expansión, añadir como `> Decisión de diseño: ...` con label explícito
```

### Regla 7 — Verificación post-fix

Tras aplicar cambios a un paso:

```
1. grep "29 filas|580 celdas|S_efectiva|FAQPage en HP|Local entity"  →  0 matches
2. wc -l antes vs después  →  diferencia explicable
3. ¿Bloque IV Fuentes intacto?  →  sí
4. ¿Numeración de outputs consistente entre §2 / §5 / §6 / §7 / §8?  →  sí
```

## Implementación

`scripts/qa-check.ts` ejecuta:
- 12 reglas QA por página
- 35 criterios de auditoría Paso 18
- 7 checks de fidelidad (grep + diff vs catálogo)

Si cualquier check falla, retorna `exit 1` y bloquea el script de deploy.

## Los 5 status de output (no son reglas, son notación)

| Status | Significado | Cuándo se usa |
|---|---|---|
| `confirmed` | valor real validado (preflight cliente, doctrina aplicada directamente) | output con valor definitivo |
| `validated` | tool externa real ejecutada y verificada | Local Pack ejecutado, KW research con datos reales |
| `⚠ inferido` | la IA estimó sin ejecutar la tool externa real | KW research no ejecutado, decidimos no abrir Maps |
| `⚠ placeholder` | input del cliente pendiente | NAP completo, fotos, web ref no entregados |
| `no aplica` | output marcado así en doctrina | 3.12, 10.3, 15.13, 16.3, 17.1 |

### Reglas operativas de los `⚠`

1. **No paralizan los Bloques 1-5.** La web sale LIVE con `⚠` posibles.
2. **Al cerrar Bloque 5 → se vuelcan al `clientes/[slug]/pendientes.md`.**
3. **El Bloque 7 (GBP) está bloqueado** hasta que `pendientes.md` esté vacío.
4. **La fuente del output no cambia** cuando hay `⚠`. Solo cambia el status. Esto preserva trazabilidad: cuando vea `⚠ inferido`, sé qué tool ejecutar.
